## Top issues (fix-first)

1. **AI Search parses "in" as state Indiana.** `q=Flemish people working in biotech in Boston` produces `city=Boston&state=IN` — the parser is reading the **preposition "in"** as the two-letter state code IN. So *any* English query of the form "<noun> in <city>" silently filters to Indiana. This is one root cause behind a lot of empty/wrong results across normal-sounding searches. Fix: state extraction must require either an explicit comma (`Boston, MA`), an uppercase 2-letter token, or a known state name — not bare lowercase "in".
2. **Search filter chips persist across queries.** After search 1 sets `sector=Biotechnology&city=Boston&state=IN`, running a different query (`KU Leuven alumni in healthcare`) keeps those chips active in URL+UI but then *also* shows AI-Enhanced Results that ignore them — chips lie about what's filtering.
3. **Stale empty-state header.** Every search shows "No people found matching the name '<query>'" line above the actual results card. Looks broken on every result page.
4. **Discovery: silent first-click failure.** When a stale `running` row exists, clicking Run Discovery kills the zombie but does NOT start a new run (only an OPTIONS preflight, no POST). User must click again, with no toast/error. Confirmed via network panel.
5. **Discovery: broken runbook reference.** Failure cards say `See docs/RUNBOOK.md entry [db_timeout] when available.` — "when available" reads like the doc doesn't exist; users follow the link to nothing.
6. **Collections suggestion crashes with raw JS error.** `Find Collection Suggestions` on a populated collection returns the literal string `Cannot read properties of undefined (reading 'rest')` in a red banner. Edge function/contract mismatch leaking through to the user.
7. **`/admin/discovery` mismatch with docs.** `ROUTES.md` says the Discovery tab includes a "held-out Discovery Eval panel" — it isn't there (and `DiscoveryEvalPanel.tsx` was deleted). Update docs or restore the panel intentionally.
8. **`/admin/growth` is missing most documented panels.** Docs promise source yield, entity pivots, geography gaps, discovery planning, reflection status, and recommended actions. Page only renders `Where to look next` reflection cards.

## Second pass (additional findings)

### AI Search shape coverage
- **`founders in Indianapolis`** → 0 results, no chips applied. Indianapolis isn't recognized as a city even when plainly stated; only the 14 seed cities are matched.
- **`Belgian founders in SF`** → 0 results; `SF`, `NYC`, `LA` abbreviations aren't expanded to their cities. Add a small alias map.
- **`Boston, MA biotech`** → 20 results in Boston/Cambridge, MA. So an explicit `City, ST` form works — confirms the Boston-IN bug is purely the bare-`in` parser.
- **`AI researchers in Atlanta`** → 2 Atlanta GA results, no Indiana misfire — so the `in`→IN bug only triggers when followed by a token that *also* exists as a city in Indiana.
- AI Search `Boston, MA biotech` returned an MIT computer-science professor among biotech results — the AI rerank loosely matches sector terms; either label "loose match" or filter harder.
- Sparkle icon at right of search bar: clicking it does nothing visible (no menu, no toggle, no tooltip) — looks like a dead control.
