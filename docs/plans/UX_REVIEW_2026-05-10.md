## Top issues (fix-first)

1. **Autofill takes ~1.1 s per keystroke.** `UnifiedSearchBar.tsx:49–104` runs `or=(name.ilike,%q%, first_name.ilike, last_name.ilike)` against the **raw `people` table** — trigram GIN indexes only exist on `people_search_documents.name_normalized` / `organization_search_documents.name_normalized`, so autofill always seq-scans. Plus the people and organization queries are awaited **sequentially**, the `locations(*)` join is included but never rendered, no `AbortController` (stale-response race), no client cache, and queries fire at `q.length === 1`. Measured ~440 ms p50 each leg, frequent 700–1500 ms cold; total stroke pipeline ≈ 1.1 s. Repoint at the trigram-indexed search-document tables (or hit `search-people` Stage-1 with `limit=8`), `Promise.all` the two queries, raise floor to 2 chars, drop the wasted join, attach `AbortController`. Estimated win: ~1100 ms → ~200 ms. Direct match for the user's "autofill very very slow" complaint.

## Root-cause investigation (follow-up pass)

Five follow-up agents (G–K) re-investigated each open question with code-grep, SQL diagnostics, and re-tests. Per-agent notes in `/tmp/uxreview/{G..K}_*.md`. The seven open root causes resolved as below, plus three new findings and three items still unverified due to a browser-tooling block.

### Why the workspace logs out (top-issues #2 and #3 — same root)

`src/lib/auth.tsx:139` and `:158` both call `await supabase.auth.signOut()` in the catch branch of `hydrateSession` whenever `loadStaffUser()` rejects. Default scope is `'global'`, so supabase-js's BroadcastChannel fires `SIGNED_OUT` on every tab on the origin. Each victim tab redirects to `/login?redirect=<its-own-pathname>` via `RequireAuth` at `auth.tsx:255–258` — that is why a tab that never visited `/admin/overview` lands at `redirect=%2Fadmin%2Foverview` (it was the URL of the tab at the moment of the broadcast, not a string built somewhere in the source).

The legacy admin slugs `/admin/{overview,discovered,agents}` do NOT call signOut themselves. They reach `<Admin>`, fire several authed queries before the redirect effect runs, and expose the same `loadStaffUser` race — so a transient failure on those routes triggers the same global broadcast. Adding a `<Navigate>` redirect for the slugs is the docs/ROUTES.md fix, but it does NOT fix the underlying signOut path; both must change.

`loadStaffUser` calls `activate_staff_user_session` (`auth.tsx:79`). The RPC fires 3–5× per page load because `auth.tsx:165–193` runs `getSession()` *and* subscribes to `onAuthStateChange` (`INITIAL_SESSION` + `TOKEN_REFRESHED` × N + focus events) with no single-flight guard, and the cache-hit branch additionally schedules a background revalidation. Any single transient failure (network jitter, refresh race, missing JWT email claim per migration L28-30, or the JWKS issue described next) of any of those 5 calls triggers the global signOut. This is why "Run Discovery silently logs out" and "visiting /admin/overview silently logs out" are the same bug — both expose a transient `loadStaffUser` failure, both bridge through the unconditional `scope:'global'` signOut.

**Root-cause-fix bundle:**
- `src/lib/auth.tsx:139` — remove the `signOut()` (keep cached profile on transient failures; do not nuke the session for anything but a confirmed *invalid* JWT).
- `src/lib/auth.tsx:158` — change to `signOut({ scope: 'local' })` so an actual logout in this tab does not broadcast to siblings.
- `src/lib/auth.tsx:165–193` — add a single-flight ref-guard (or promise cache keyed by user id) around `loadStaffUser` so `INITIAL_SESSION` + `TOKEN_REFRESHED` collapse to one call per session.
- `src/App.tsx` — add explicit `<Navigate to="/admin/discovery" replace>` aliases for `/admin/overview`, `/admin/discovered`, `/admin/agents`, `/contacts/new` *before* `<RequireAuth>` so legacy bookmarks short-circuit before the auth gate runs.
- New finding from agent J: `RequireAuth` at `auth.tsx:230` is *also* logging "Maximum update depth exceeded" via `Navigate → RequireAuth → AuthProvider`, suggesting the AuthProvider `useMemo` value churns (deps include `refreshStaffUser`/`signOut`) or `<Navigate replace />` receives a fresh `to=` every render. Stabilize the memo value (extract callbacks with `useCallback`, freeze the `value` object across renders unless user/session actually changes) when the signOut fix is in.

### Why `suggest-people` 401s while `search-people` works (top-issue #6)

Both functions import the same `requireStaffRole` from `supabase/functions/_shared/auth.ts:88`, which calls `supabase.auth.getUser(jwt)` against the project's Auth API. There is no code-level asymmetry. The 401 comes from a **stale per-worker JWKS cache** inside the Supabase Edge runtime: the project uses ES256 asymmetric tokens (per `supabase/config.toml:3–6`), the function imports `npm:@supabase/supabase-js@2` (floating tag), and warm Deno workers cache the JWKS. A worker started before the last Auth signing-key rotation cannot verify newly-issued tokens — the kid `b81269f1-…` is not in the cached set.

`search-people` was redeployed recently (the new `supabase/functions/search-people/rerank.ts` is in the working tree), so its workers cold-started after the rotation and fetched a fresh JWKS. `suggest-people` was not redeployed and its warm worker is stuck on the old key set.

**Root-cause fix:**
- Pin `npm:@supabase/supabase-js@2.57.4` (or whichever stable version matches the client) in every edge function's import map; do not float `@2`.
- Redeploy *all* edge functions in one pass (`supabase functions deploy --project-ref ofzuhajxwxggybkuzefq`) so every worker cold-starts and re-fetches JWKS.
- Add a 401-on-getUser retry that forces a JWKS refresh (`supabase.auth.getUser(jwt)` does not expose this directly; either switch to a verifier that supports kid-miss → fetch, or add a watchdog that recreates the inner client on auth_failed).
- This will recur every Auth signing-key rotation until pinned + redeployed; document in `docs/RUNBOOK.md` (the new file already in the working tree).

### Why the `'rest'` banner appears (top-issue #7) — NOT a client bug

`extractEdgeError` is a red herring. The "Cannot read properties of undefined (reading 'rest')" error is thrown **server-side** inside `supabase/functions/suggest-people/index.ts:202–212`'s `callUntypedRpc` helper, which detaches `rpc` from its bound `supabase` client. PostgREST's internal `this.rest` is then `undefined` and the helper crashes before the response leaves the function. `supabase/functions/search-people/index.ts:262–278` has the same helper but already fixed — its comment block literally documents the exact error string. The fix never propagated to `suggest-people`.

**Root-cause fix:** in `supabase/functions/suggest-people/index.ts`, invoke as `client.rpc(...)` (preserve `this`), exactly as `search-people` does. One-line change. After this fix, when ROOT-CAUSE-2 (JWKS) is unresolved, users will at least see the real `auth_failed` error code and can self-diagnose.

### Why the verification queue stays stuck (top-issues #4 + #5)

Six rows have `verification_status='queued'` with `verification_run_id=NULL` — confirmed via SQL: they were **never claimed**. One contact (Sofie Peeters) is `status='approved' + verification_status='queued'`, which is hidden by `DiscoveredContactsPanel.tsx:1142–1147`'s strict `.eq('status','pending')` filter — that single row is the entire `DB-shows-2 / UI-shows-1` discrepancy.

`enqueueVerificationBatch` in `supabase/functions/agent-scheduler/index.ts:836–843` silently resets rows to `queued, verification_run_id=null` when `forwardedAuth` is missing (i.e. when housekeeping ran without an Authorization header to forward). It writes nothing to `agent_runs.error_message` and returns ok-success — so the System card flips to `idle, Last success 5s` and the panel banner says "Housekeeping completed" while the queue is exactly where it was.

**Root-cause fix:**
- `agent-scheduler/index.ts:836` — when `forwardedAuth` is missing, mark the parent `agent_runs` row `failed` with `error_kind='auth_missing'` and a clear `error_message`. Do NOT silently reset rows. The System card stale-failure suppression then surfaces the issue correctly.
- `agent-scheduler/index.ts:859–873` (the dispatch-catch) — same: persist the failure to `agent_runs.error_message` before any reset. Add a `verification_attempts` counter on the discovered row so a row that has failed N times is escalated rather than silently re-queued forever.
- `DiscoveredContactsPanel.tsx:1142–1147` — drop the `eq('status','pending')` filter; key the queue load on `verification_status IN ('queued','verifying','verified')`. Alternative: write a one-shot migration to bring `status` back in sync with `verification_status` and document the contract that `status` is now derived.
- `DiscoveredContactsPanel.tsx:1577` and `:2023` — change the default fallback from `verification_status ?? 'verified'` to `?? 'queued'`; a NULL must NOT bypass the verify gate.

### Why realtime never connects on `/admin/verification`

Publication setup is correct: `supabase/migrations/20260508000006_verify_before_promote.sql:108–118` adds both `discovered_contacts` and `discovered_organizations` to `supabase_realtime`. The bug is in the client: `DiscoveredContactsPanel.tsx:1266`'s `.subscribe()` has **no status callback**, so `SUBSCRIBED` / `CHANNEL_ERROR` / `TIMED_OUT` are all invisible. The likely failure is RLS — SELECT on these tables requires `has_staff_role('editor')`, and realtime's RLS evaluator may not resolve helper functions that depend on auth context the same way the REST gateway does.

**Root-cause fix:**
- `DiscoveredContactsPanel.tsx:1266` — pass a `(status, err) => console.debug('verification realtime', status, err)` callback to `.subscribe()` so the failure mode is observable.
- Verify in browser console after the fix; if `CHANNEL_ERROR` appears, simplify the RLS policy on the realtime publication path (e.g. allow SELECT for any authenticated user with a non-NULL `staff_users` row) or add a polling fallback (refetch every 15 s while a queued/verifying row is in view).
